# Story 1.7: Newsletter capture (global) + Newsletter page

**Status**: Draft
**Epic**: 1 — Hub Foundation
**Depends on**: 1.1
**Build-order note**: despite the numeric ID, this story must be built **before**
Story 1.6 (Home page) — 1.6's AC1 requires the shared capture component this story
produces. See 1.6's "Depends on" line.

## Story

As any visitor, I want a newsletter signup available everywhere and a dedicated
`/newsletter/` page, so that the site has one owned, compounding conversion channel
(AP-5).

## Acceptance criteria

1. A single shared newsletter-capture component renders in: the global footer (all
   pages), home's primary placement (Story 1.6), `/newsletter/`, and the standard
   end-of-entity CTA block (used by project/publication detail pages, Stories 1.3/
   1.5) — one implementation, reused everywhere (FR6, spec §8.1).
2. The component is a static-safe embed or link-out form to an external ESP — no
   self-hosted form handling, no email address stored in this repo or its deployment
   (NFR2, NFR3, §8.1).
3. If the ESP embed fails to load, the component degrades to a plain link (NFR6).
4. `/newsletter/` renders: pitch (what subscribers get, honest low-frequency cadence,
   sample topics from focus areas), signup form, and an empty (Phase-3-deferred)
   archive slot that renders nothing until Story 3.6 (spec §7.3 "Newsletter", §9
   cadence-honesty requirement).
5. No page anywhere commits to a publishing frequency in copy (§9 anti-treadmill
   requirement) — content review, not just code review, gates this.

## Tasks / subtasks

- [ ] Choose/confirm ESP integration shape (embed vs. link-out) — actual provider is
      deferred (spec D-3); build against a generic embed contract so swapping
      providers later doesn't touch every page.
- [ ] Build the shared capture component and the standard end-of-entity CTA block
      (used by Stories 1.3, 1.5, and later 2.3/3.2).
- [ ] Build `/newsletter/` page.

## Dev notes

- ESP selection (spec D-3) is out of scope for this story — build the integration
  point generically (a single component + a single config value) so the eventual
  provider choice is a config change, not a rewrite.
