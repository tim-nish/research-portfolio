# Story 1.1: Content schema & build-time validation pipeline

**Status**: Draft
**Epic**: 1 — Hub Foundation
**Depends on**: none

## Story

As the site's implementer, I want a markdown+frontmatter content-ingestion layer with
schema validation for all six content types, so that every later page-building story
can trust the registry, and the build fails fast on any malformed record (FR2/AC-4).

## Acceptance criteria

1. A schema is defined for each of `profile`, `project`, `product`, `publication`,
   `article`, `newsletter-issue`, matching `docs/website-architecture-spec.md` §6.2
   field-for-field (required/optional, enum values, nesting) — including the two
   additive fields from `docs/architecture.md` §9 (`product.unlisted`,
   `product.featured`, `publication.featured`).
2. A build-time script parses every file under `content/`, validates it against its
   type's schema, and fails the build (non-zero exit, readable error naming file +
   field) on: missing required field, unknown enum value, duplicate `slug` within a
   type, or an unresolvable `related.*` slug reference.
3. A `mode: external` article record with a body, or a `mode: canonical` record
   without one, fails the build (schema for `article` land now even though its pages
   ship in Epic 2, per Epic 1's "in scope" note).
4. The validator runs as part of `npm run build` (per `CLAUDE.md`'s existing
   `tsc -b && vite build` pipeline) — no separate manual step to remember.
5. The parsed, validated output is an in-memory typed registry available to the
   static-generation step, with inverse relations computed once per build (FR9).
6. Unit-level coverage (or equivalent fixture-driven check) exists for each failure
   mode in AC2/AC3 above, proving the gate actually rejects bad input, not just
   accepts good input.

## Tasks / subtasks

- [ ] Add `gray-matter` (or equivalent) + a schema-validation library (e.g. `zod`).
- [ ] Write one schema module per content type under `src/content/schema/`.
- [ ] Write `scripts/validate-content` that walks `content/`, parses, validates,
      resolves `related.*`, checks slug uniqueness, and computes inverse relations.
- [ ] Wire the script into the `build` npm script ahead of `vite build`.
- [ ] Add fixture records (good + deliberately bad, one per failure mode) under a
      test-only content path excluded from the live registry, and a check that
      asserts each bad fixture fails validation with a legible error.
