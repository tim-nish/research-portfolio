# Story 5.4: Pre-commit publication lint — stale or hand-edited projections block publish

Canonical discussion record: https://github.com/tim-nish/website/issues/68
(part of umbrella #64 — `docs/article-publishing-spec.md` publication rules;
owner decision record 2026-07-18). On conflict, the issue wins.

As the site owner, I want a pre-commit lint over `content/articles/`, so
that stale or hand-edited projections and stray pipeline output are blocked
before they can be published.

## Acceptance criteria

1. **Given** a projection with a frontmatter schema violation, **when** a
   commit is attempted, **then** the commit is blocked (sharing 5.1's
   validator).
2. **Given** a projection whose declared `source` pin cannot be what
   generated the committed body (e.g. body edited in this repo after
   generation), **when** the lint runs, **then** it is flagged as
   hand-edited — the authoring side wins and the projection here is the
   defect.
3. **Given** generated proposal/intermediate output anywhere inside the site
   tree, **when** the lint runs, **then** it is rejected.
4. **Given** the lint script, **when** invoked standalone, **then**
   `npm run lint:publication` runs it, and it is wired as the pre-commit
   hook.
