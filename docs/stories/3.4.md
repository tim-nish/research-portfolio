# Story 3.4: Sunset & lifecycle behaviors

**Status**: Done
**Epic**: 3 — Products & Validation
**Depends on**: 3.1, 3.2, 3.3

## Story

As the site owner, I want the product-sunset lifecycle fully implemented as a
first-class state, so that killing a product is a designed, transparent event, not a
broken page (spec §6.4).

## Acceptance criteria

1. A `sunset` product's landing page (Story 3.2) shows the sunset note, optional
   successor pointer, and newsletter capture in place of the normal CTA/body blocks —
   confirmed against the Story 3.1 fixture.
2. A `sunset` product moves to the collapsed "Retired" section of `/products/`
   (Story 3.3) with its sunset date shown.
3. The product's URL never 404s and never redirects away — it remains at
   `/products/<slug>/` permanently (AP-3), only its rendered state changes.
4. This is exercised end-to-end: create a fixture or real `sunset` record, confirm
   both the detail page and the index page render its sunset behavior correctly
   (AC-8).

## Completion notes

Verification pass only — no gaps found, no production code changed.

- **AC-1** (sunset landing page): `ProductDetailPage` (`src/pages/ProductDetailPage.tsx`)
  already swaps the CTA/body/platform blocks for the sunset note + optional
  successor link when `status === "sunset"`, and `ReducedChromeLayout` always
  renders the footer `NewsletterCapture` regardless of status — so the newsletter
  capture remains as the page's sole remaining action on a sunset page. Covered by
  `src/pages/__tests__/ProductDetailPage.test.tsx`'s "sunset lifecycle (AC3)" suite,
  including the successor-link present/absent cases.
- **AC-2** (Retired section on `/products/`): `groupProductsForIndex`
  (`src/content/productView.ts`) always buckets `sunset` records into `retired`
  regardless of `unlisted`, and `ProductsIndexPage` renders them in the collapsed
  `<details class="products-retired">` block with `sunset.date`. Covered by
  `src/content/__tests__/productView.test.ts`'s "always includes sunset products in
  retired" test.
- **AC-3** (URL stability): `generateProductDetailPages` in
  `scripts/generate-static.tsx` iterates every product record unconditionally and
  writes to `/products/<slug>/` — sunset is a render-time branch, not a routing
  branch. Confirmed via a clean `npm run build`: the sunset fixture is emitted as a
  real static page (`Generated /products/placeholder-sunset-product/`, distinct
  from the separately-logged "Generated redirect stub" lines) and appears in
  `sitemap.xml`; `content/redirects.yml` has no entry for it.
- **AC-4** (end-to-end fixture): `content/products/placeholder-sunset-product.md`
  (added in Story 3.1) already exercises this — verified the built
  `dist/products/placeholder-sunset-product/index.html` renders the sunset block
  and `dist/products/index.html` lists it under "Retired (1) ... retired 2026-06".

All 87 existing tests pass unchanged; no new tests were needed since Stories
3.1–3.3 already established permanent regression coverage for every AC here.
