# Story 1.3: Project content type, Projects index, Project detail

**Status**: Draft
**Epic**: 1 — Hub Foundation
**Depends on**: 1.1

## Story

As a fellow researcher (P1, journey J1), I want to browse all projects and open a
project's detail page, so that I can see what it is, whether it's credible, and where
the code/paper live.

## Acceptance criteria

1. `project` records validate against spec §6.2's schema (`slug`, `title`, `kind`,
   `status`, `summary`, `started`, `links`, optional `featured`/`topics`/`related`/
   `citation`).
2. `/projects/` renders per spec §7.3: intro line from profile focus areas;
   active/maintained projects as cards (featured first, then `started` descending);
   an archive section for `archived` projects, visually de-emphasized or collapsed.
3. `/projects/<slug>/` renders, in order: title + kind/status line + started date;
   links row (primary action zone, above the fold); optional body; optional citation
   block (copyable BibTeX); optional related-entities section; standard CTA block.
4. `archived` status renders its §6.4 banner and drops from featured/home surfaces
   while remaining in the index.
5. SEO metadata (FR7) and JSON-LD present per type (no dataset-bearing type here
   unless the project is a benchmark with dataset links — see Story 1.4 overlap).
6. Broken `related.*` references fail the build (already covered by 1.1, verified
   here against real project records).

## Tasks / subtasks
