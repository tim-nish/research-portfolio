# Story 1.10: Redirect map + custom 404 page

**Status**: Draft
**Epic**: 1 — Hub Foundation
**Depends on**: 1.1

## Story

As the site owner, I want a declarative redirect mechanism and a custom 404 page, so
that renamed/retired URLs never break (AP-3) and lost visitors can still find their
way to the hub.

## Acceptance criteria

1. `content/redirects.yml` (or equivalent) declares `source → destination` pairs.
2. At build time, a static stub page is generated at each `source` path per
   `docs/architecture.md` §7 (D-9): `rel=canonical` to destination, meta-refresh to
   destination, a plain fallback link — readable and functional with JS disabled.
3. At least one redirect entry exists and is verified end-to-end post-build (source
   path resolves to destination) — satisfies AC-3's "verified with a test redirect."
4. `/404` renders the identity line and links to the main indexes (spec §7.1, §7.3
   note under Home is not applicable here — this is its own page per the URL
   taxonomy).
5. Trailing-slash policy is decided once (implementer's choice per spec §7.1) and
   applied consistently across every route generated by every other Epic 1 story —
   this story is where that policy is implemented as a build-wide redirect/rewrite
   rule, not per-page.

## Tasks / subtasks

- [ ] Implement the redirect-stub generator (build-time step).
- [ ] Add one real or placeholder redirect entry and verify it resolves.
- [ ] Build the `/404` page.
- [ ] Decide and document the trailing-slash policy; implement as a single,
      cross-cutting rule.

## Dev notes

- See `docs/architecture.md` §7 (D-9) for the full rationale on why GitHub Pages
  needs the meta-refresh-stub approach rather than a true HTTP redirect.
