# Story 3.5: Validation-speed proof run

**Status**: Done
**Epic**: 3 — Products & Validation
**Depends on**: 3.1, 3.2, 3.3

## Story

As the site owner, I want to measure how long it actually takes to publish a new
`validating` product, so that AP-12's ½-day-validation-gate premise (the site as its
instrument) is proven, not assumed.

## Acceptance criteria

1. A new `validating` product is published (real product from the backlog, e.g. one
   named in spec Appendix A's future-records list, or a fixture if none is ready) by
   creating exactly one content record file — no code change (AC-2).
2. The elapsed content-authoring time (writing the record, not including any one-time
   setup) is measured and is under 30 minutes (AC-10).
3. The result (pass/fail, actual time) is recorded (e.g. as a note in this story file
   or a short entry in `docs/architecture.md`'s log) so the claim is falsifiable, not
   just asserted.

## Tasks / subtasks

- [x] Pick the product to validate (owner's call — do not invent product content).
- [x] Time the authoring of its record file, start to publish-ready.
- [x] Record the result against AC-10.

## Dev notes

- This story only has a "done" state once a real timing measurement exists — a
  hypothetical estimate does not satisfy AC-10.

## Completion notes

No real product from the backlog was ready to publish, so — per the owner's explicit
choice — this is run against a fixture record, `content/products/validation-speed-proof.md`,
clearly labeled `[Fixture]` and not a real product (spec's own AC-1 explicitly allows
"a fixture if none is ready").

**Timing result (AC-10): PASS.**

- Start: 2026-07-07T05:35:43Z (first write of the record file)
- End: 2026-07-07T05:35:59Z (record complete, schema-valid, publish-ready)
- Elapsed: well under the 30-minute bar.

**Caveat on what this measures**: this timing reflects AI-assisted authoring (this
session), not the site owner's own unassisted authoring speed. It falsifies the
*mechanical* half of AP-12's claim — that the pipeline itself imposes no per-record
code change and no structural friction (AC-2, confirmed below) — but a fully
representative proof of the ½-day-validation-gate premise still wants one real timed
run by the owner against an actual backlog product. Recommend re-running this
measurement (or appending to it) the next time a real `validating` product is
authored by hand.

**AC-2 (no code change) confirmed**: the fixture was published by adding exactly one
file, `content/products/validation-speed-proof.md`. `npm run validate:content`,
`npm test -- --run` (87/87 passing, unchanged), and a clean `npm run build` all
succeeded with zero changes to any `src/`, `scripts/`, or schema file — the new
record was picked up automatically by the existing content-registry loader and
appeared as `Generated /products/validation-speed-proof/` in the build log.
