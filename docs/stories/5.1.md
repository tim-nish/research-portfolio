# Story 5.1: Projection content type + frontmatter validation

Canonical discussion record: https://github.com/tim-nish/website/issues/65
(part of umbrella #64 — `docs/article-publishing-spec.md` §2). On conflict,
the issue wins.

As the site owner, I want a site-canonical projection content type
(`content/articles/<slug>.md`) with build-time frontmatter validation, so
that published projections carry a verifiable birth record and malformed
files can never reach the built site.

## Acceptance criteria

1. **Given** a projection file at `content/articles/<slug>.md` with required
   keys `title`, `slug`, `source` (`<authoring-repo>@<commit>` pin),
   `variant: site`, `language`, `published`, `generated_by`, **when** the
   build runs, **then** the file validates and is available to later
   rendering stories.
2. **Given** a projection file missing or mistyping any required key,
   **when** the build runs, **then** the build fails naming the offending
   file and key.
3. **Given** a valid projection, **when** page or index titles are derived,
   **then** they come from frontmatter `title`, never from the body (title
   authority amendment).
4. **Given** the existing Epic 2 external article records (#23), **when**
   this schema is added, **then** external records keep their existing shape
   and continue to validate — this story extends, never replaces, and
   renders no pages.
